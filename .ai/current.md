# Claude Code 세션 연속성 워크플로우 설계

> 세션 종료/중단 후 작업을 이어가기 위한 워크플로우 설계

---

## 문제 인식

| 문제 | 상황 |
|------|------|
| 세션 종료 후 AI가 컨텍스트를 잃음 | exit, crash, 세션 만료 |
| 여러 기능 동시 작업 시 AI가 맥락 혼동 | 여러 기능 병렬 작업 |
| context 꽉 찰 때 자동 관리 안 됨 | 긴 작업 세션 |
| 프로젝트 커지면 .ai/ 작업 내용 누적 | 장기 프로젝트 |

---

## 설계 원칙

- **가볍게, 파일 하나로** — 무거운 자동화 대신 단순한 패턴
- **한 번에 하나의 기능** — 여러 기능 동시 작업 시 git worktree로 격리
- **규모에 따라 스케일** — 단순 작업이면 20줄, 복잡하면 500줄, 같은 파일

---

## 콘텐츠 유형

### 1. 작업 문서 (.ai/current.md) — 확정

**하나의 파일이 규모에 따라 스케일:**

| 작업 규모 | current.md 크기 | 포함 내용 |
|----------|----------------|----------|
| 단순 (버그, 작은 기능) | ~20줄 | 목표 + 진행 + 다음 할 일 |
| 복잡 (멀티 세션 기능) | ~500줄 | 스펙 + 설계 결정 + 구현 단계 + 진행 |

- 세션 시작 시 **전체 읽기** (500줄도 Claude 컨텍스트 대비 작은 양)
- frontmatter 없음 (작업 문서에 불필요한 ceremony 제거)
- 작업 완료 시 삭제 or 패턴으로 승격

**템플릿**: `.ai/templates/simple-task.md`, `complex-task.md`

**라이프사이클**:
```
작업 시작      → current.md 생성 (템플릿 기반)
세션 중단      → /save로 현재 상태 저장
세션 재개      → current.md 읽고 이어서
작업 완료      → 삭제 or patterns/로 승격
```

### 2. 패턴 레퍼런스 (.ai/patterns/) — 확정

**프로젝트에서 실제로 쓴 구현 패턴** (범용 지식은 obsidian).

**핵심 특징**:
- 선택적 저장 (가치 있는 작업만)
- 비슷한 작업 시 재활용 (A 작업 참고해서 E 작업 진행)
- 반복하면서 점진적 개선 (A+E 경험으로 패턴 업데이트)

**패턴 매칭 방식**:
- 사용자가 직접 지정: "config-driven-table 패턴 참고해"
- 또는 패턴 인덱스(README.md) 보고 찾기

**파일 구조**:
```
.ai/patterns/
├── README.md                    ← 인덱스 (패턴 요약 테이블)
├── config-driven-table.md
└── crud-page.md
```

**인덱스 (README.md) 포맷**:
```markdown
# 패턴 인덱스

| 패턴 | 설명 | 적용 사례 |
|------|------|----------|
| config-driven-table | 메뉴별 필터/컬럼이 다른 목록 페이지 | 구매 상담 |
| crud-page | 기본 CRUD 목록+상세 | 사용자 관리 |
```

**패턴 파일 내부 구조**:
```markdown
---
updated: 2026-02-20
---

# {패턴명}

## 언제 쓰는지
{이런 상황에서 참고}

## 핵심 구조
{파일 트리, 주요 타입}

## 핵심 결정
{왜 이렇게 했는지}

## 코드 예시
{실제 코드 패턴}

## 변경 메모
- 2026-02-20: 초기 작성
```

**모노레포에서의 접근**:
- 패턴은 **모노레포 루트** `.ai/patterns/`에 저장
- 루트 CLAUDE.md에 경로 명시 → 서비스에서 실행해도 Claude가 계층적 CLAUDE.md 읽으므로 경로 인지 가능

**배움/지식 vs 패턴 구분**:
```
obsidian:  "React Query 캐시 동작 원리"        ← 범용 지식
patterns:  "우리 프로젝트 필터+테이블 레시피"    ← 프로젝트 구현 패턴
```

### 3. 저장 위치 분류 — 확정

```
.ai/current.md        ← 활성 작업 상태 (일시적, 덮어쓰기)
.ai/patterns/         ← 프로젝트 구현 패턴 (장기, 선택적 누적, 점진적 개선)
.ai/templates/        ← 작업 문서 템플릿
obsidian-note/        ← 배움/지식 (영구, 개인)
{project}/CLAUDE.md   ← 프로젝트 규칙 (영구, 프로젝트)
~/.claude/MEMORY.md   ← 글로벌 패턴 (영구, 200줄 한도)
```

## git worktree = 컨텍스트 격리 — 확정

```
~/project/              ← main
~/project-auth/         ← feature/auth (별도 디렉토리)
~/project-payment/      ← feature/payment (별도 디렉토리)

→ 각 worktree가 독립적 컨텍스트
→ Claude 열면 해당 worktree만 인식
→ 멀티 기능 컨텍스트 혼동 없음
```

## Claude memory 한계 인식

| 상황 | 신뢰도 |
|------|--------|
| 단일 세션, 단일 기능 | 높음 |
| 단일 세션, 여러 기능 | context 크기 따라 |
| **멀티 세션, 여러 기능** | 낮음 (git worktree로 해결) |

```
MEMORY.md       → 세션 시작 시 자동 로드, 200줄 한도
다른 memory 파일 → 자동 로드 안 됨, 명시적으로 읽어야 함
```
