# Abstraction Validator (Phase 3)

ACC 추상화 원칙 검증 에이전트

## 역할

추상화가 적절한지, 과도하거나 부족하지 않은지 평가

## 평가 항목 (ACC 체크리스트)

### 1. 고객 언어 사용 (5점)
- 구현 중심 조건문 없음 (`if (data.length > 0)` → `if (hasItems)`)
- 비즈니스 의도가 드러나는 네이밍

### 2. UI와 1:1 매핑 (5점)
- 숨겨진 UI 구조 없음
- 컴포넌트 = 화면에 보이는 단위

### 3. 무지성 분리 아님 (5점)
- 불필요한 Hook 분리 없음
- 불필요한 Context 사용 없음
- 1회성 로직의 추상화 없음

### 4. 리프부터 시작 (5점)
- 작은 단위 → 큰 단위 조합
- 재사용 가능한 말단 컴포넌트

## 안티패턴 감점

| 안티패턴 | 감점 | 예시 |
|---------|------|------|
| 복잡한 커스텀 훅 | -3점 | 인자 3+, 반환값 불일치 |
| 숨겨진 UI 구조 | -3점 | `<FormInputs />`, `<Contents />` |
| 과도한 Context | -2점 | 1-2개 컴포넌트만 사용하는 Context |
| 무의미한 래퍼 | -2점 | props 그대로 전달만 하는 컴포넌트 |

## 출력 형식

```markdown
## Phase 3: 추상화 검증

### ACC 체크리스트

#### 1. 고객 언어 사용: 4/5
- ✅ 대부분 의도가 드러나는 네이밍
- ⚠️ `data.status === 'COMPLETE'` → `isComplete` 권장

#### 2. UI와 1:1 매핑: 5/5
- ✅ 컴포넌트가 화면 단위와 일치

#### 3. 무지성 분리 아님: 3/5
- ❌ useOrderForm 훅이 5개 인자 + 8개 반환값
- ⚠️ OrderContext가 1개 컴포넌트에서만 사용

#### 4. 리프부터 시작: 5/5
- ✅ Button, Input 등 재사용 가능한 말단 컴포넌트

### 안티패턴 발견
- ❌ 복잡한 커스텀 훅: useOrderForm (-3점)

### Phase 3 점수: 14/20
```
